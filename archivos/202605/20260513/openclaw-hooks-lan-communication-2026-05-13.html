Resumen Ejecutivo

Investigación sobre mecanismos de comunicación entre agentes OpenClaw (TARS, CASE, KIPP) dentro de una red local. Se identificaron dos categorías de hooks y múltiples estrategias de implementación.

Arquitectura de Hooks en OpenClaw

1. Internal Hooks (Eventos del Gateway)

Scripts instalados en el workspace que responden a eventos del ciclo de vida del sistema.

Eventos disponibles:

Evento Descripción
command:new Cuando se ejecuta /new
command:reset Cuando se ejecuta /reset
gateway:startup Al iniciar el Gateway
agent:bootstrap Durante bootstrap del agente
session:compact:before Antes de compactación de sesión

Estructura de ejemplo:

hooks/
  └── session-memory/
      ├── HOOK.md (metadatos)
      └── handler.ts (lógica)

Comandos CLI:

openclaw hooks list          # Listar hooks disponibles
openclaw hooks enable <name> # Habilitar hook
openclaw hooks info <name>   # Ver detalles

Documentación oficial: - https://docs.openclaw.ai/automation/hooks - https://docs.openclaw.ai/cli/hooks

2. Plugin Hooks (Extensibilidad In-Process)

Hooks programáticos para plugins que necesitan interceptar eventos del sistema.

Catálogo completo de hooks:

Categoría Hooks Disponibles
Agent turn before_model_resolve, before_prompt_build, before_agent_reply, agent_end
Tools before_tool_call, after_tool_call, tool_result_persist
Messages message_received, message_sending, message_sent
Sessions session_start, session_end, before_compaction
Subagents subagent_spawning, subagent_spawned, subagent_ended
Gateway gateway_start, gateway_stop, cron_changed

Ejemplo de registro de plugin:

import { definePluginEntry } from "openclaw/plugin-sdk/plugin-entry";

export default definePluginEntry({
  id: "my-plugin",
  register(api) {
    api.on("before_tool_call", async (event) => {
      if (event.toolName !== "exec") return;
      return {
        requireApproval: {
          title: "Ejecutar comando",
          description: `Allow: ${event.params.command}`,
        },
      };
    });
  },
});

Documentación: - https://docs.openclaw.ai/plugins/hooks - https://docs.openclaw.ai/plugins/sdk-runtime

Webhooks para Comunicación Externa

Plugin de Webhooks (HTTP Entrypoints)

OpenClaw incluye un plugin de webhooks que expone endpoints HTTP autenticados para triggerar TaskFlows desde sistemas externos.

Configuración en gateway:

{
  plugins: {
    entries: {
      webhooks: {
        enabled: true,
        config: {
          routes: {
            "lan-bridge": {
              path: "/plugins/webhooks/lan-bridge",
              "method": "POST",
              "auth": "bearer",
              "taskFlow": {
                "name": "lan-command-handler",
                "payload": {
                  "message": "Comando recibido desde LAN"
                }
              }
            },
            "kipp-command": {
              path: "/plugins/webhooks/kipp-command",
              "method": "POST",
              "auth": "bearer",
              "taskFlow": {
                "name": "kipp-executor",
                "payload": {
                  "command": "{{body.command}}",
                  "source": "{{body.source}}"
                }
              }
            }
          }
        }
      }
    }
  }
}

Implementación para Comunicación LAN

Estrategia Recomendada: Webhooks + TaskFlow

Arquitectura:

┌─────────────┐      HTTP POST       ┌─────────────┐
│   CASE/TARS │ ───────────────────► │    KIPP     │
│  (OpenClaw) │  (webhook endpoint)  │  (OpenClaw) │
│             │                      │             │
│  Envía:     │                      │  Recibe:    │
│  - command  │                      │  - command  │
│  - payload  │                      │  - payload  │
│  - source   │                      │  - source   │
└─────────────┘                      └─────────────┘

Paso 1: Configurar Webhook en KIPP

En el gateway de KIPP, habilitar el plugin de webhooks:

// ~/.openclaw/gateway/config.json
{
  plugins: {
    entries: {
      webhooks: {
        enabled: true,
        config: {
          routes: {
            "kipp-exec": {
              path: "/plugins/webhooks/kipp-exec",
              method: "POST",
              auth: "bearer",
              tokenEnv: "LAN_HOOK_TOKEN",
              taskFlow: {
                name: "kipp-command-executor",
                payload: {
                  kind: "systemEvent",
                  text: "Comando LAN recibido: {{body.command}}"
                }
              }
            }
          }
        }
      }
    }
  }
}

Paso 2: Crear TaskFlow Receptor

# TaskFlow: kipp-command-executor

**Trigger:** Webhook POST a /plugins/webhooks/kipp-exec

**Payload esperado:**
```json
{
  "command": "exec",
  "params": {
    "command": "ls -la"
  },
  "source": "CASE",
  "callback": "http://case-local-ip:PORT/plugins/webhooks/case-response"
}

Acciones: 1. Validar token de autenticación 2. Parsear comando y parámetros 3. Ejecutar con exec tool 4. Enviar resultado al callback URL


### Paso 3: Script de Envío desde CASE/TARS

```javascript
// lan-command-sender.js
const fetch = require('node-fetch');

async function sendCommand(targetAgent, command, params) {
  const response = await fetch(`http://${targetAgent}-ip:8383/plugins/webhooks/${targetAgent}-exec`, {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'Authorization': `Bearer ${process.env.LAN_HOOK_TOKEN}`
    },
    body: JSON.stringify({
      command,
      params,
      source: 'CASE',
      timestamp: Date.now()
    })
  });
  
  return response.json();
}

// Uso
sendCommand('kipp', 'exec', { command: 'git status' });

Consideraciones de Seguridad

🔐 Autenticación

  1. Token compartido en .env de todos los agentes:

    LAN_HOOK_TOKEN=super-secret-token-cambia-este-valor

  2. Validación de origen - Verificar campo source en payload

  3. HTTPS recomendado - Usar certificados locales o tunnel inverso

🛡️ Autorización

  • Whitelist de comandos permitidos por agente
  • Rate limiting para evitar abuso
  • Logging de todos los comandos recibidos/ejecutados

📋 Logging

Alternativas de Implementación

Opción A: Shared Firestore (Recomendada para producción)

Usar Firebase Firestore como cola de comandos compartida:

CASE → Escribe en firestore.comandos/{id} → KIPP escucha con onSnapshot → Ejecuta

Ventajas: - ✅ No requiere exposición HTTP directa - ✅ Cola persistente de comandos - ✅ Historial auditado - ✅ Funciona detrás de NAT

Ver investigación complementaria: firebase-firestore-event-listener-2026-05-13.md

Opción B: Socket.io en LAN

Crear servidor Socket.io que escuche en la red local:

// kipp-socket-server.js
const io = require('socket.io')(3000, {
  cors: { origin: '*' }
});

io.on('connection', (socket) => {
  socket.on('command', async (data) => {
    // Ejecutar comando
    socket.emit('response', { result });
  });
});

Opción C: SSH Directo

Si los agentes tienen SSH habilitado:

ssh kipp@kipp-local-ip "openclaw run --command 'git status'"
